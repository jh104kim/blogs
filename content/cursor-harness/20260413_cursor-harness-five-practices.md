**커서(Cursor)** 는 AI 코딩 에이전트를 강하게 만들어 주는 IDE입니다.  
**하네스 엔지니어링**은 AI에게 명령만 던지는 것이 아니라, **컨텍스트(정보)·스킬(능력)·툴(도구)** 를 미리 설계해 에이전트가 오래 안정적으로 움직이게 하는 기술입니다.

아래 5가지는 **Rules, Skills, Hooks, MCP, 초기화·진행 추적** 등 커서 생태계와 맞닿은 실전 패턴입니다. 각각 **무엇을 하는지 → 왜 좋은지 → 어떻게 설정하는지** 순으로 정리했습니다. (플랜·기능에 따라 UI·경로는 버전마다 다를 수 있습니다.)

---

### 1. Rules로 프로젝트 전체 규칙(컨텍스트) 미리 알려주기

(항상 켜져 있는 ‘프로젝트 설명서’ 만들기)

- **무엇을 하나요?** AI가 매번 “TypeScript만 써야 해, 폴더 구조는 이렇게야”를 반복 설명하지 않게 합니다.
- **왜 좋은가요?** 프로젝트 스타일을 잃지 않고 일관된 코드를 기대하기 쉽습니다. 초보자가 먼저 시도하기 좋은 기본 하네스입니다.
- **커서에서 어떻게 하나요?**
  1. 프로젝트 루트에 `.cursor/rules/` 폴더를 만듭니다.
  2. 예: `coding-style.md` 파일에 규칙을 적습니다.

```markdown
# 프로젝트 규칙
- TypeScript만 사용 (JavaScript 금지)
- 컴포넌트는 components/ 폴더에 PascalCase로 만들기
- 항상 npm run lint 후 수정
```

  3. 저장하면 에이전트가 해당 규칙을 참고하도록 구성할 수 있습니다. (Cursor Rules 문서를 함께 확인하세요.)

- **활용한 요소**: Context (Rules = 지속 컨텍스트)

**초보자 팁:** “AI가 이상한 코드를 자꾸 짜요?” → Rules 파일 하나부터 만들어 보세요.

---

### 2. Skills로 특정 작업 ‘스킬’ 추가하기

(AI에게 “이건 네가 잘하는 일이야”라고 가르치기)

- **무엇을 하나요?** 필요할 때만 불러오는 특수 워크플로를 정의합니다. 항상 켜진 Rules와 달리, 필요 시에만 로드하는 식으로 토큰을 아낄 수 있습니다.
- **왜 좋은가요?** “React 컴포넌트 만들어줘” 같은 요청에 맞춰 베스트 프랙티스를 반복 적용하기 쉽습니다.
- **커서에서 어떻게 하나요?**
  1. 예: `.cursor/skills/react-component/` 폴더를 만듭니다.
  2. `SKILL.md` 예시:

```yaml
---
name: react-component
description: React 컴포넌트 만들기 (Tailwind + TypeScript)
---

# React 컴포넌트 제작 가이드
- 파일명: PascalCase.tsx
- Tailwind 클래스 사용
- Storybook 예시도 함께 만들어줘
```

  3. 채팅에서 슬래시 커맨드나 에이전트가 스킬을 선택하도록 팀 규칙을 맞춥니다.

- **활용한 요소**: Skill + Context (SKILL.md 내용)

**초보자 팁:** 공개 스킬 예시를 참고해 `SKILL.md`를 복사·수정해 써 보세요.

### 3. Hooks로 ‘자동 수정 루프’ 만들기

(AI가 테스트 실패하면 스스로 고치게 하기)

- **무엇을 하나요?** 작업이 멈췄을 때 테스트를 다시 돌리거나, 실패 시 후속 지시를 주는 흐름을 자동화합니다.
- **왜 좋은가요?** 한 번에 완벽하지 않아도 **검증 → 수정** 루프로 수렴시키기 쉽습니다.
- **커서에서 어떻게 하나요?**
  1. `.cursor/hooks.json` 예시:

```json
{
  "version": 1,
  "hooks": {
    "stop": [{ "command": "bun run .cursor/hooks/grind.ts" }]
  }
}
```

  2. `.cursor/hooks/grind.ts` 등에서 터미널·테스트 결과에 따라 후속 메시지를 반환하는 로직을 둡니다. (실제 구현은 프로젝트 스크립트에 맞게 작성합니다.)

```ts
// 예: 테스트 실패 시 에이전트에게 후속 지시를 돌려주는 훅 (의사코드)
// const failed = await runTests();
// if (failed) return { followup_message: "테스트 다시 돌리고 실패 원인 수정해." };
```

- **활용한 요소**: Tool(터미널) + Feedback Loop(Hooks)

**초보자 팁:** 처음에는 “테스트 모두 통과할 때까지 계속 작업해” 한 줄 프롬프트만으로도 비슷한 효과를 볼 수 있습니다.

### 4. MCP Tool로 외부 서비스 연결하기

(AI에게 GitHub, Figma, DB 등 ‘손’을 더해주기)

- **무엇을 하나요?** PR 생성, 이슈 조회, 디자인 참조 등 외부 시스템과 에이전트를 연결합니다.
- **왜 좋은가요?** 코드 생성을 넘어 **배포·협업 파이프라인**에 가깝게 일하게 할 수 있습니다.
- **커서에서 어떻게 하나요?**
  1. Cursor 설정에서 MCP 서버를 추가합니다.
  2. 예: 사용자 설정의 `mcp.json`에 서버 정의 (이름·커맨드는 공식 문서·각 MCP 패키지 안내를 따릅니다).

```json
{
  "mcpServers": {
    "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] }
  }
}
```

  3. 채팅에서 “PR 만들어줘”처럼 자연어로 호출합니다. (인증·토큰은 반드시 안전하게 설정하세요.)

- **활용한 요소**: Tool (MCP)

**초보자 팁:** 팀에서 이미 쓰는 도구(GitHub, DB 등)용 MCP부터 하나만 연결해 보세요.

### 5. Initializer + Progress Tracking으로 장기 프로젝트 관리

(AI가 며칠씩 일해도 안 무너지게 하기)

- **무엇을 하나요?** 시작 시 할 일 목록(`features.json` 등)과 진행 로그(`progress.md` 등)를 두어, 컨텍스트가 끊겨도 이어 갈 수 있게 합니다.
- **왜 좋은가요?** 대형 작업에서 **상태가 파일로 남아** 세션 간 연속성이 좋아집니다.
- **커서에서 어떻게 하나요?**
  1. 루트에 `AGENTS.md` 또는 `CLAUDE.md`에 초기화·진행 규칙을 적습니다.

```markdown
# 초기화 작업
1. features.json 만들고 할 일 목록 작성
2. progress.md에 현재 상태 기록
3. 항상 features.json 보고 다음 작업 선택
```

  2. 첫 대화에서 “프로젝트 초기화해줘”로 파일 생성을 맡깁니다.
  3. 병렬 에이전트·브랜치 전략은 팀 규모에 맞게 조정합니다.

- **활용한 요소**: Context(AGENTS.md) + Feedback(progress)

**초보자 팁:** 작은 토이 프로젝트부터 목록·로그 파일 패턴을 익히면 큰 저장소로 확장하기 쉽습니다.

---

이 다섯 가지를 쌓으면 Cursor 에이전트가 **단순 도우미에서 협업 동료에 가깝게** 느껴질 때가 많습니다. 추천 순서는 **Rules → Skills → Hooks**이고, 프로젝트가 커지면 **MCP·진행 추적**을 더하면 됩니다.

필요하면 각 사례별 **파일 템플릿**이나 스크린샷 기준 설명을 더 붙일 수 있습니다. Cursor를 열고 작은 규칙 파일 하나부터 실습해 보세요.
